# 棣友足球队内部 PWA 网站 Spec

## Why
棣友足球队沉淀了 11 年的照片视频、文章诗词与比赛记录，需要一个以手机端为主、可安装为 PWA 的内部网站进行系统化归档、检索与展示，并支持基于本地知识库的大模型辅助年会策划。

## What Changes
- 新建前后端分离项目：前端 Vue + TypeScript（移动优先、响应式、PWA），后端 Node.js + TypeScript（模块化 API 服务）
- 建立统一设计系统：设计令牌（颜色/字号/间距/圆角/阴影）、基础组件库（按钮/卡片/表单/下拉菜单/列表/空状态）
- 建立数据库与媒体文件存储方案：PostgreSQL 作为主库；照片/视频等媒体文件落本机磁盘；元数据入库
- 提供成员账号体系与权限：每位队友独立账号，支持角色/权限（至少：管理员、成员）
- 提供内容域能力：照片/视频、文章/诗词、比赛结果、队员集锦、年度归档与时间轴浏览、资料上传与维护
- 提供本地知识库与年会策划智能能力：面向“年会活动策划”提供可配置 Skill（提示词模板 + RAG 检索 + 结构化输出），第一阶段接入千问 Qwen
- 全站多语言：至少支持中/英的 UI 文案切换（内容数据本身不强制翻译）
- **BREAKING**：无（新项目）

## Impact
- Affected specs: 移动端信息架构、设计系统、后端模块与 API、数据模型与迁移、PWA 能力、权限模型、知识库与大模型接入
- Affected code: 新建代码库（frontend/ 与 backend/），数据库迁移与配置（PostgreSQL），文件存储目录与访问策略

## ADDED Requirements
### Requirement: 技术与架构基线
系统 SHALL 使用 TypeScript；前端 SHALL 使用 Vue（Vue 3 生态）并实现移动优先的响应式 UI；后端 SHALL 使用 Node.js（TypeScript）并以模块化方式提供 REST API；前后端 SHALL 通过明确的 API 契约解耦。

#### Scenario: 基线满足
- **WHEN** 启动前端与后端服务
- **THEN** 前端通过配置的 API Base URL 访问后端；无硬编码 mock 数据；页面在移动端正常交互

### Requirement: PWA 安装与离线基线
系统 SHALL 支持以 PWA 方式安装到手机桌面，并具备基础离线能力（至少：App Shell、静态资源缓存；不强制缓存媒体大文件）。

#### Scenario: 安装成功
- **WHEN** 用户在移动端浏览器打开站点并选择“添加到主屏幕/安装”
- **THEN** 站点可被安装，并以独立应用形态启动

### Requirement: 账号体系与权限
系统 SHALL 提供成员账号体系，每位队友一个账号；系统 SHALL 支持至少两类角色：管理员、成员；系统 SHALL 对“上传/编辑/删除”操作进行权限校验。

#### Scenario: 登录与受保护操作
- **WHEN** 成员完成登录
- **THEN** 可访问其权限允许的页面与 API
- **WHEN** 未登录用户访问受保护资源
- **THEN** 被拒绝并引导至登录流程

### Requirement: 内容域 - 照片/视频
系统 SHALL 支持上传、浏览、筛选、按时间线聚合展示照片/视频；第一阶段 SHALL 支持“手动标注人物标签”，以实现按队员筛选查看其相关照片/视频。

#### Scenario: 按人物查看
- **WHEN** 用户为媒体条目添加人物标签
- **THEN** 在“按人物”入口可查看该人物关联的全部媒体

### Requirement: 内容域 - 文集/诗集
系统 SHALL 支持录入/上传文章与诗词（文本与可选附件），支持按作者（队员）筛选与搜索，支持作品切换浏览。

#### Scenario: 按作者检索作品
- **WHEN** 用户在文集/诗集页面输入人物姓名并搜索
- **THEN** 展示该作者的全部文字作品

### Requirement: 内容域 - 比赛记录
系统 SHALL 支持记录并展示历次比赛结果（日期、红蓝队、比分、参赛队员、MVP），并提供按年份/时间筛选。

#### Scenario: 浏览比赛结果
- **WHEN** 用户进入比赛记录列表并选择年份
- **THEN** 展示该年份下的比赛列表与详情

### Requirement: 内容域 - 队员集锦与个人档案
系统 SHALL 提供个人页面（空白模板）与单人详情页；个人页面顶部 SHALL 提供三个下拉菜单：XX 号家庭、红队、蓝队；用户选中后才显示对应人物/家庭信息；系统 SHALL 支持为人物关联媒体、文字作品与比赛参与信息。

#### Scenario: 下拉筛选人物
- **WHEN** 用户在个人页面选择“红队”并选中某位队员
- **THEN** 展示其个人档案与关联内容入口

### Requirement: 年度时间轴与年度页
系统 SHALL 提供“历史页面”，在移动端以可纵向折线/波浪线表现的时间轴呈现年份节点（2015–2026 + 更多）；点击年份 SHALL 进入年度页面，展示该年度的照片/视频、文章/诗词、比赛结果、年内重大聚会条目。

#### Scenario: 年份导航
- **WHEN** 用户点击历史页面的 2015 节点
- **THEN** 进入 2015 年年度页面并展示该年度可用内容（为空则展示空状态）

### Requirement: 上传资料与内容维护
系统 SHALL 提供上传资料页面，允许管理员/授权成员创建/维护人物档案，并上传照片/视频、录入文章/诗词、录入比赛记录等；系统 SHALL 不内置任何 mock 数据；空数据场景 SHALL 有明确空状态。

#### Scenario: 上传与入库
- **WHEN** 授权用户上传媒体并填写元数据（时间、人物标签等）
- **THEN** 媒体文件被保存到本机存储目录；元数据写入 PostgreSQL；前端可浏览到新内容

### Requirement: 本地知识库与 Qwen 年会策划 Skill
系统 SHALL 提供本地知识库：可上传/录入与年会相关的资料（规则、往届方案、预算模板等）；系统 SHALL 集成千问 Qwen 作为第一阶段 LLM 提供方；系统 SHALL 提供“年会策划 Skill”能力，基于知识库检索结果生成结构化方案（至少包含：目标与主题、流程与主持词、物料清单、人员分工、预算与风险预案）。

#### Scenario: 生成年会方案
- **WHEN** 用户在年会策划界面输入约束（人数、预算、时间、地点、风格）
- **THEN** 系统结合知识库内容与提示词模板生成完整年会活动方案，并以可复制/导出的结构呈现

### Requirement: 依赖版本策略
系统 SHALL 在项目初始化时使用官方稳定版依赖（与对应生态的最新稳定版本对齐）；系统 SHALL 使用锁文件（lockfile）固定依赖解析结果；系统 SHALL 提供升级策略与约束（例如：按月升级、CI 校验）。

#### Scenario: 依赖可追溯
- **WHEN** 安装依赖并提交 lockfile
- **THEN** 不同环境可复现一致的依赖树

## MODIFIED Requirements
无（新项目）
