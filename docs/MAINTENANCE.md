# 棣友足球队档案站 — 维护与升级计划

本文档面向项目负责人和后续开发者，说明项目上线后应如何持续维护、保证安全、并平稳升级。

## 1. 维护目标

1. **安全**：防止数据泄露、未授权访问、文件上传漏洞、JWT 泄露等。
2. **稳定**：保证 NAS 部署后 7×24 小时可用，定期备份可恢复。
3. **可持续**：依赖不过度落后，文档与代码保持一致，新功能可低成本接入。

## 2. 当前已知风险（截至 2026-06-18）

运行 `npm audit` 后发现项目存在多项依赖漏洞，风险等级包括：

- **Critical**：`fast-jwt` — JWT auth bypass
- **High**：`axios`、`exifreader`、`@babel/*`、`esbuild` 等
- **Moderate**：`dompurify`、`brace-expansion` 等

> 这些漏洞主要来自开发工具链和前端依赖，**在局域网 / 私人 NAS 场景下直接暴露面较小**，但仍应尽快修复。

## 3. 维护节奏（建议）

### 每周（Weekly）

| 任务 | 操作 | 耗时 |
|------|------|------|
| 检查服务状态 | 登录 NAS，查看 Docker 容器是否运行 | 5 分钟 |
| 查看备份 | 确认 `scripts/backup.sh` 生成的备份文件存在且非空 | 5 分钟 |
| 日志巡检 | 查看 `docker logs diyou-backend` 是否有异常报错或大量失败登录 | 10 分钟 |
| 磁盘空间 | 检查 NAS 存储是否充足，清理过期备份 | 5 分钟 |

### 每月（Monthly）

| 任务 | 操作 | 耗时 |
|------|------|------|
| 依赖安全扫描 | 运行 `npm audit`，处理可自动修复的漏洞 | 30 分钟 |
| 补丁/小版本升级 | 升级 `patch` 和 `minor` 版本依赖，运行测试 | 1–2 小时 |
| 备份恢复演练 | 在测试环境用 `scripts/restore.sh` 恢复一次备份 | 30 分钟 |
| 密码与密钥检查 | 确认 `.env` 中的 `JWT_SECRET` 强度高、未泄露；必要时轮换 | 15 分钟 |
| 用户权限审计 | 检查管理员账号数量、是否有离职队员账号未清理 | 10 分钟 |

### 每季度（Quarterly）

| 任务 | 操作 | 耗时 |
|------|------|------|
| 大版本依赖评估 | 检查 Vue、Vite、Fastify、Prisma 等大版本更新，决定是否升级 | 2–4 小时 |
| Node.js / Docker 镜像更新 | 更新 `node:xx-alpine` 基础镜像，重新构建并测试 | 1–2 小时 |
| 安全加固复查 | 检查 CORS、_rate limit_、文件上传白名单、HTTPS 证书有效期 | 1 小时 |
| 漏洞手动修复 | 对 `npm audit` 无法自动修复的高危漏洞，评估替代方案或手动补丁 | 2–4 小时 |
| 数据归档 | 将历史年份数据导出存档，减轻线上数据库体积 | 1 小时 |

### 每年（Yearly）

| 任务 | 操作 | 耗时 |
|------|------|------|
| 架构回顾 | 评估是否需要拆分服务、引入 CDN、对象存储等 | 半天 |
| 灾难恢复演练 | 完整模拟 NAS 故障，按文档在新设备上重建服务 | 半天 |
| 合规与隐私 | 确认无敏感个人信息违规存储；清理过期日志 | 1 小时 |

## 4. 依赖升级策略

### 版本号规则

- `patch`（如 `1.2.3` → `1.2.4`）：通常只修 bug，可自动升级。
- `minor`（如 `1.2.x` → `1.3.0`）：新增功能，可能不破坏 API，升级后需跑测试。
- `major`（如 `1.x` → `2.0`）：可能有破坏性变更，升级前需阅读 CHANGELOG，安排专门迭代。

### 升级流程

1. 在独立分支执行：
   ```bash
   npm update -w backend -w frontend
   npm audit fix
   ```
2. 运行完整测试：
   ```bash
   npm run -w backend test
   npm run -w frontend build
   npm run -w backend build
   ```
3. 本地/测试环境手动验证关键路径（登录、上传媒体、创建大事记、批量导入）。
4. 合并到 `main`，触发 CI，部署到 NAS。

## 5. 安全维护清单

### 5.1 运行环境

- [ ] NAS 只开放必要的端口（HTTP/HTTPS，必要时 SSH）。
- [ ] 路由器/防火墙关闭 3001 等内部端口的公网映射。
- [ ] 使用 Caddy 自动 HTTPS，Let's Encrypt 证书自动续期。
- [ ] 若需公网访问，优先考虑内网穿透或 VPN，而非直接暴露端口。

### 5.2 应用层

- [ ] `JWT_SECRET` 长度 ≥ 32 字节随机字符串，定期轮换。
- [ ] 管理员密码强度高，不与其他平台共用。
- [ ] 文件上传限制类型（图片/视频/文本），后端已做白名单和路径安全处理。
- [ ] 输入数据使用 Zod 校验，Markdown 内容经 DOMPurify 清理。
- [ ] 限流（rate-limit）已开启，防止暴力破解。
- [ ] 后端日志不输出敏感信息（密码、JWT、文件路径）。

### 5.3 容器与镜像

- [ ] 定期更新基础镜像 `node:xx-alpine`。
- [ ] 生产镜像使用非 root 用户运行（可进一步优化 Dockerfile）。
- [ ] 镜像构建后扫描漏洞：`docker scan` 或 Trivy。

## 6. 备份与灾难恢复

项目已提供：

- `scripts/backup.sh`：备份 SQLite 数据库和媒体文件。
- `scripts/restore.sh`：从备份恢复。
- `scripts/smoke-test.sh`：部署后基础健康检查。

### 建议

1. **备份频率**：至少每天一次，NAS 可设置 cron。
2. **保留策略**：本地保留 7 天滚动备份；重要节点（比赛、年会后）手动另存一份。
3. **异地备份**：每周将备份同步到云盘或另一台设备。
4. **恢复演练**：每季度在测试 NAS/电脑上按 `RESTORE.md` 恢复一次。

## 7. 监控与告警

### 基础监控

- 健康检查端点：`GET /health`（已存在）。
- 容器状态：NAS 的 Docker 面板或 `docker compose ps`。
- 日志：`docker logs --tail 100 -f diyou-backend`。

### 进阶方案（可选）

- 使用 Uptime Kuma 监控 `https://nas地址` 是否可达。
- 使用 Promtail + Grafana 收集日志并告警异常登录、上传失败。
- 设置磁盘空间告警（< 20% 时通知）。

## 8. 推荐自动化

### GitHub 侧

- 开启 **Dependabot alerts** 和 **Dependabot security updates**。
- 开启 **CodeQL** 静态安全扫描。
- 要求所有代码合并前 CI 通过（测试 + 构建）。

### NAS 侧

- 使用 `cron` 每天执行备份脚本。
- 使用 Watchtower 自动拉取并更新镜像（已配置在 `docker-compose.yml` 中，默认未启用）。

## 10. 技术债务跟踪

以下问题建议纳入后续迭代：

- [ ] 后端 `any` 类型较多，可逐步替换为严格的 Prisma / Zod 类型。
- [ ] 前端部分页面硬编码 API 路径，应统一通过 service 层。
- [ ] 图片/视频存储目前直接放在 NAS 本地，数据量大时可迁移到对象存储（MinIO / S3）。
- [ ] 批量导入、智能导入等功能可补充前端单元测试。

**维护原则**：小步快跑、先安全后功能、每次变更都有测试和回滚方案。
